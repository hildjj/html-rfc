3. HTML Format

The format specified here is a subset of HTML5, deemed to be widely-implemented by common browsers at the time that the specification was created, likely to continue to be widely-implemented for in the future, and unlikely to cause security issues.

3.1. Syntax

The following rules SHALL be enforced before submittal.

  • The HTML source MUST be encoded as UTF-8, as specified in RFC 3629.
  • The HTML source MUST be formatted in the manner of well-formed XML, with all elements start tags having matching end tags (or <element /> for empty elements), with all elements properly nested.
  • HTML SHALL be pretty-printed, using spaces (not tabs) for indentation.
  • Single quotes (U+0027 APOSTROPHE: ') MUST be used to quote attribute values. Unquoted attribute values are NOT ALLOWED.
  • Two (2) spaces used for each level of indentation
  • Each text-containing element such as headings (<h1>-<h6>), paragraphs (<p>), or list items (<li>), will be rendered as a single line without wrapping.

Note: none of these rules affect the rendered output of the HTML, but are intended to increase the chance that multiple tools that process the format will generate identical syntax. In turn, this will make difference tools that operate on the HTML source easier to write.

3.2. Basic Structure

3.2.1. DOCTYPE

The DOCTYPE of the document MUST be "html", which declares that the document is compliant with HTML5. For example, the document will start with exactly this string:

<!DOCTYPE html>

3.2.2. Charset Declaration

In order to be correctly processed by browsers that load the HTML using a mechanism that does not provide a valid MIME content-type or charset, the HTML <header> element MUST contain a <meta> element, with the attributes http-equiv='Content-Type' and content='text/html; charset=utf-8'. This will look like:

<meta http-equiv='Content-Type' content='text/html; charset=utf-8' />

3.2.3. Style

The header SHOULD contain an embedded stylesheet. The styles in the stylesheet are to be set consistently between documents by the RFC editor, according to the best practices of the day. The stylesheet MUST NOT be considered as normative information. To ensure consistent formatting, individual style attributes SHOULD NOT be used in the main portion of the document source except in highly exception circumstances; each use MUST be individually justified.

3.2.4. Comments

HTML comments MAY be used, but MUST NOT contain normative information. One example is to clarify particular choices in the HTML format. Example:

<!-- Automatically generated: do not modify -->

3.2.5. Sections

Each section of the document SHALL be formatted as a <div> tag, with a class attribute with value "section". A document-unique, id attribute SHOULD be assigned to each section <div>. The id MAY be human-readable or generated. Each section <div> MUST contain a header tag (<h2>-<h6>) of the appropriate depth, with top-level sections getting an <h2> tag, and each nested section getting the next higher header level. If more than five levels of headers are required, <h6> MUST be used for each deeper-nested section. However, nesting sections more than five levels deep is NOT RECOMMENDED.

The text in each header tag MUST begin with the section number. Section numbers MUST begin at "1.", and MUST increment by one for each successive section at the same level. Subsections MUST be numbered by appending the subsection number to the parent section number.

Within a section, each "normal" paragraph MUST be surrounded by a <p> element. For example:

<div class='section' id='example'>
  <h2>1. Example Section</h2>
  <p id='example-p-1'>This is a description of the example</p>
  <div class='section' id='nested'>
    <h3>1.1. Nested Section</h3>
    <p id='nested-p-1'>This is a description of the nested section.</p>
    <p id='nested-p-2'>This is the second description paragraph.</p>
  </div>
</div>

3.2.6. Paragraphs

Paragraphs MUST be contained in a section div. A document-unique, id attribute SHOULD be assigned to each <p>. The id will usually be machine-generated, but MAY be human-readable if desired.

It is RECOMMENDED that each paragraph be kept relatively small compared to a "page" in previous RFC formats, so that references to each paragraph are at least as valuable as page references have been in previous formats.

5.3. Images

Include an img tag with a src attribute, which specifies a file name relative to the source document, or a http(s) URL for an image of type png, jpg, or gif. The image will be retrieved and converted to a data: URI as specified in RFC2397. The alt attribute is REQUIRED, and MUST be a complete, accessible description for any diagram intended to be normative.

<img src='nit-flow.png' alt='A description of the diagram'/>

5.4. Sequence Diagrams

Include an image tag with class='sequence', where the alt text is the WebSequenceDiagrams.com source for the diagram. A reminder: in the alt text the author MUST escape all less-than (< to &lt;), greater-than (> to &gt;), and ampersand (& to &amp;) characters, in the same manner normally required by HTML.

<img class='sequence' alt='
title Authentication Sequence
Alice-&gt;Bob: Authentication Request
note right of Bob: Bob thinks about it
Bob-&gt;Alice: Authentication Response' />
